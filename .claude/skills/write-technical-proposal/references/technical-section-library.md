# Technical Section Library

Use this as a section menu, not as a rigid table of contents. Follow the tender's required order when it provides one.

## Common Technical Volume Structure

1. Technical response summary
2. Understanding of project background, objectives, scope, and constraints
3. Overall solution design
4. System architecture or technical route
5. Functional response by module or requirement group
6. Non-functional response: performance, security, reliability, compatibility, scalability, usability
7. Integration and data/interface plan
8. Implementation plan and milestone schedule
9. Project organization and staffing
10. Quality assurance and testing plan
11. Delivery, deployment, migration, cutover, and acceptance support
12. Training plan
13. Operations, maintenance, after-sales, and service response
14. Risk management and contingency plan
15. Documentation and deliverables
16. Technical deviation/clarification list

## Writing Patterns

### Requirement Response

Use when the tender contains detailed clauses.

Structure:

1. "针对[requirement], 我方响应如下:"
2. Direct compliance statement.
3. Method or configuration.
4. Deliverable or verification method.
5. Evidence or reference to provided material.

### Solution Design

Structure:

1. Design goals tied to the tender.
2. Architecture/components.
3. Data flow or process flow.
4. Key technical measures.
5. Operations and maintenance considerations.

### Implementation Plan

Structure:

1. Stage names and objectives.
2. Activities and outputs.
3. Owner/team.
4. Time or dependency.
5. Acceptance checkpoint.

### Service Plan

Structure:

1. Service scope.
2. Response mechanism.
3. Support channels and staffing.
4. SLA only if confirmed by source material.
5. Reporting, review, and continuous improvement.

## Useful Evidence Types

- Product datasheets or technical white papers
- Prior project cases with similar scope
- Test reports and acceptance reports
- Certifications and authorization letters
- Architecture diagrams, screenshots, configuration examples
- Team resumes and role descriptions
- Standard operating procedures, emergency plans, service records

Never invent evidence. Mark missing evidence as a gap and keep it visible in the review notes.
